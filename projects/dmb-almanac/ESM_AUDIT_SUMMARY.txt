ISSUE #1: Missing Error Handling on Dynamic Imports
────────────────────────────────────────────────────
Severity: WARNING (Medium)
Status: Fixable in < 10 minutes

Affected files (6 locations):
  • src/lib/stores/data.ts:59
  • src/lib/stores/dexie.ts:74
  • src/lib/utils/appBadge.ts:59
  • src/lib/db/dexie/queries.ts:1292, 1333, 1374, 1401, 1427

Problem:
  Dynamic imports lack try-catch blocks. If a module fails to load,
  the error will propagate uncaught, potentially breaking runtime.

Current pattern:
  const { loadInitialData } = await import('$db/dexie/data-loader');

Recommended pattern:
  try {
    const { loadInitialData } = await import('$db/dexie/data-loader');
    // Use loadInitialData
  } catch (error) {
    console.error('[Module] Failed to load:', error);
    // Provide fallback or re-throw
  }

Impact: MEDIUM
- Could cause runtime crashes if modules fail
- No performance overhead from fix
- Improves robustness

Fix time: ~10 minutes (all locations)


ISSUE #2: Inconsistent WASM Import Paths
─────────────────────────────────────────
Severity: WARNING (Low)
Status: Fixable in < 5 minutes

File with issue:
  src/lib/wasm/transform.ts:102

Problem:
  Uses relative path instead of path alias

Current code:
  const wasm = await import('../../../wasm/dmb-transform/pkg/dmb_transform.js');

Recommended code:
  const wasm = await import('$wasm/dmb-transform/pkg/dmb_transform.js');

Why standardize:
  • Consistency with rest of codebase
  • More maintainable if structure changes
  • Shorter, cleaner code
  • Matches patterns in advanced-modules.ts

Impact: LOW
- Both patterns work correctly
- Standardization improves code quality
- No functional change

Fix time: ~2 minutes

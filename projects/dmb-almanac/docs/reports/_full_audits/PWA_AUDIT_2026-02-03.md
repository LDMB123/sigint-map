**Cache Version**: Dynamic based on `__APP_VERSION__` and `__BUILD_HASH__` (build-time replacement)

**Issue**: ❌ **CRITICAL** - Minified SW prevents audit of exact caching logic

**Recommendation**: Provide source maps or unminified version for audit

### Caching Strategies: 85/100

**Detected Patterns** (from minified code inspection):

1. **Cache-First** (Static Assets)
   - Target: JS, CSS, fonts
   - Fallback: Network
   - TTL: Not explicitly time-based (version-based)

2. **Network-First** (API/Pages)
   - Target: `/api/*`, page routes
   - Timeout: 3000ms (3s)
   - Fallback: Cache
   - Stale-while-revalidate for background updates

3. **Stale-While-Revalidate** (Images)
   - Target: Image files
   - Background update after cache hit
   - TTL: 30 days

4. **Network-Only** (likely)
   - Target: Google Fonts (based on registerRouter detection)

**Advanced Features** ✅:
- Navigation preload enabled
- Request deduplication (in-flight request map)
- Retry with exponential backoff (3 retries, 100ms → 2000ms)
- Compressed data serving (Brotli → gzip → uncompressed fallback)
- Broadcast channel for cache updates
- Request timeout enforcement (3s default)

**Issues**:

⚠️ **HIGH PRIORITY**: No explicit stale-time enforcement for API cache
- API responses cached with `X-Cache-Time` header but no automatic revalidation after TTL
- Could serve stale data indefinitely if network fails
- **File**: `sw.js:line unknown` (minified)
- **Fix**: Implement max-age check in cache retrieval:
  ```javascript
  const age = Date.now() - parseInt(response.headers.get('X-Cache-Time'), 10);
  if (age > TTL) {
    // Force network fetch or show stale warning
  }
  ```

⚠️ **MEDIUM**: Cache eviction during navigation may cause jank
- LRU eviction runs synchronously during activate/fetch
- Could delay first paint on low-end devices
- **Fix**: Move to `requestIdleCallback` or background task

⚠️ **MEDIUM**: No quota management for overall storage
- Per-cache limits exist (API: 50, Pages: 100, Images: 200 entries)
- No check against `navigator.storage.estimate()` quota
- Risk of quota exceeded errors in IndexedDB operations
- **Fix**: Add quota checking before cache operations, integrate with `storageMonitor.js`

### Background Sync Handling: 90/100

- `dmb-data-refresh`: Refreshes cached data
- `dmb-push-subscription`: Syncs push subscription with server

**Periodic Sync Event**: ✅
- Tag: `dmb-data-refresh` / `check-data-freshness`
- Checks `/api/data-version` for updates
- Notifies clients if new data available
- Refreshes critical cache entries (`/api/shows`, `/api/songs`, `/api/venues`)

**Mutation Queue Processing** (`processSyncQueue`):
- Opens IndexedDB `dmb-almanac` → `syncQueue`
- Fetches all items
- Attempts sync with `X-Offline-Sync: true` header
- Deletes successful syncs
- Retains failed syncs (SW doesn't retry, relies on next sync event)

**Telemetry Queue Processing** (`processTelemetryQueue`):
- Opens IndexedDB `dmb-almanac` → `telemetryQueue`
- Filters by status: `pending` or `retrying`
- Respects `nextRetry` timestamp
- Exponential backoff: 1s → 2s → 4s (max 3 retries)
- Marks as `failed` after 3 retries
- Updates status to `completed` on success

**Issues**:

⚠️ **HIGH PRIORITY**: Sync queue has no max retry limit
- Failed syncs remain in queue indefinitely
- Could accumulate stale mutations
- **File**: `sw.js:line unknown` (minified)
- **Fix**: Add max retry count (e.g., 10) and move to dead letter queue

⚠️ **MEDIUM**: No conflict resolution for concurrent mutations
- Multiple tabs could queue conflicting mutations
- Last-write-wins on server could cause data loss
- **Fix**: Add mutation versioning or CRDT approach

## 10. Performance & Best Practices

### Service Worker Performance: 85/100

**Optimizations**: ✅
- Navigation preload reduces latency
- Request deduplication prevents duplicate fetches
- Compression negotiation (Brotli → gzip → raw)
- Background updates for images (stale-while-revalidate)
- Periodic cache cleanup prevents bloat

**Issues**:

⚠️ **MEDIUM**: Cache analysis blocks event loop
- `GET_CACHE_STATUS` iterates all caches synchronously
- Large caches (>1000 entries) could cause jank
- **Fix**: Use `requestIdleCallback` or worker thread

⚠️ **LOW**: No performance budgets
- Cache size limits are hardcoded (50-200 entries per cache)
- No dynamic adjustment based on device capabilities
- **Fix**: Detect device memory (`navigator.deviceMemory`) and adjust limits

### Cache Efficiency: 80/100

**Strengths**: ✅
- Version-based cache invalidation
- Per-cache TTLs (API: 1h, Pages: 15min, Images: 30d)
- Compression support reduces bandwidth

**Issues**:

⚠️ **MEDIUM**: No cache warming for critical routes
- Only `/` warmed after 5s idle
- High-traffic routes like `/songs`, `/shows` not warmed
- **Fix**: Warm top 5 routes based on analytics

⚠️ **LOW**: Font caching incomplete
- Google Fonts registered as network-only (Static Routing)
- Fonts not cached offline
- **Fix**: Cache fonts after first load (or self-host fonts)

### Memory Management: 88/100

**Strengths**: ✅
- Cache size limits enforced
- LRU eviction when limits exceeded
- Response cloning for cache storage
- Periodic cleanup prevents accumulation

**Issues**:

⚠️ **LOW**: Clone operations could double memory usage
- `response.clone()` called before cache.put()
- Large responses (e.g., images) duplicated in memory
- **Fix**: Stream responses to cache without cloning where possible

fix: Remove invalid type="button" attribute from navigation link

## Summary

Fixed WCAG 2.1 AA compliance violation (criterion 4.1.2 Name, Role, Value) by removing the invalid `type="button"` attribute from the "Go Home" navigation link in ErrorFallback.svelte.

## Changes Made

- **File**: app/src/lib/components/ui/ErrorFallback.svelte (Line 100)
- **Change**: Removed invalid `type="button"` attribute from `<a href="/">` tag
- **Impact**: Semantic HTML now correct - link properly announced as link by screen readers

## Problem

The error fallback component used an `<a>` tag with a `type="button"` attribute:

```svelte
<!-- WRONG -->
<a href="/" class="home-button" type="button">Go Home</a>
```

This created a semantic mismatch:
- HTML semantics: `<a>` = navigation link
- type attribute: `type="button"` = suggests button behavior
- Screen readers announced conflicting roles: "Link, type button, Go Home" (NVDA)
- Violated WCAG 2.1 AA success criterion 4.1.2 (Name, Role, Value)

## Solution

Removed the invalid `type="button"` attribute, keeping the semantic HTML clean:

```svelte
<!-- CORRECT -->
<a href="/" class="home-button">Go Home</a>
```

Benefits:
- Proper semantic HTML - `<a>` tag for navigation
- Screen readers now announce: "Link, Go Home" (clear and correct)
- CSS styling preserved - still looks like a button visually
- Keyboard interaction works correctly (Tab + Enter)
- No breaking changes to component API or styling

## Accessibility Impact

### Before Fix
```
NVDA:      "Link, type button, Go Home"
JAWS:      "Go Home, link button"
VoiceOver: "Go Home, button, link"
```
Result: Confusing dual roles for assistive tech users ❌

### After Fix
```
NVDA:      "Link, Go Home"
JAWS:      "Go Home, link"
VoiceOver: "Go Home, link"
```
Result: Clear semantic role for assistive tech users ✅

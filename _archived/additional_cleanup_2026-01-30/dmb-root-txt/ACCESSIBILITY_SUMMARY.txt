ISSUE #1: FOCUS OUTLINE REMOVAL
════════════════════════════════════════════════════════════════════════════

  WCAG 2.4.7: Focus Visible (Level AA)

  Problem:
  • Form inputs had outline: none, hiding focus from keyboard users
  • Screen reader users and keyboard-only users couldn't see focused field

  Solution:
  • Removed outline: none from :focus rule in src/app.css
  • Preserved :focus-visible with 2px solid outline
  • Box-shadow provides visual feedback from design system

  Result:
  ✓ Keyboard users see 2px amber outline (#d97706) on focused fields
  ✓ Focus indicators visible in all browsers
  ✓ Complies with WCAG 2.4.7


ISSUE #2 & #3: TABLE KEYBOARD NAVIGATION
════════════════════════════════════════════════════════════════════════════

  WCAG 2.1.1: Keyboard (Level A)

  Problems:
  • Sortable table headers not activable via keyboard
  • Clickable rows not activable via keyboard
  • Users had to use mouse for all table interactions

  Solution:
  • Added handleHeaderKeydown() function for headers
  • Added handleRowKeydown() function for rows
  • Both listen for Enter and Space keys
  • Same handlers as click events

  Result:
  ✓ Tab to sortable header, press Enter/Space to sort
  ✓ Tab to clickable row, press Enter/Space to activate
  ✓ No keyboard traps - can Tab away
  ✓ Complies with WCAG 2.1.1


ISSUE #4: TAB ARROW KEY NAVIGATION
════════════════════════════════════════════════════════════════════════════

  WAI-ARIA: Tab Pattern (Authoring Practices Guide)
  WCAG 2.1.1: Keyboard (Level A)

  Problems:
  • Tab navigation only worked with Tab key (inefficient)
  • Each tab required individual Tab press
  • Standard tab pattern uses arrow keys

  Solution:
  • Added handleTabKeydown() function
  • Supports ArrowRight/Left, ArrowUp/Down, Home, End keys
  • Implements roving tabindex pattern
  • Only active tab has tabindex="0"

  Result:
  ✓ ArrowRight/Down moves to next tab
  ✓ ArrowLeft/Up moves to previous tab
  ✓ Home jumps to first tab
  ✓ End jumps to last tab
  ✓ Tab key exits tab list to content
  ✓ Complies with WAI-ARIA tab pattern
  ✓ Complies with WCAG 2.1.1


ISSUE #5: TAB PANEL ASSOCIATIONS
════════════════════════════════════════════════════════════════════════════

  WCAG 1.3.1: Info and Relationships (Level A)
  WCAG 4.1.2: Name, Role, Value (Level A)

  Problem:
  • Tab panels weren't programmatically associated with buttons
  • Screen reader users didn't know which panel belonged to which tab
  • Semantic relationships not exposed to assistive technology

  Solution:
  • Added id attribute to tab buttons (transitions-tab, etc.)
  • Added aria-controls on buttons pointing to panels
  • Added role="tabpanel" to panels
  • Added aria-labelledby on panels pointing to buttons

  Result:
  ✓ Screen readers announce tab-panel associations
  ✓ Semantic structure exposed to assistive technology
  ✓ Complies with WCAG 1.3.1 and 4.1.2

MEMORY GROWTH PATTERNS
================================================================================

UNDER NORMAL LOAD (10K songs + 500K setlist entries):
  Initial: 50MB JS heap + 16MB WASM
  After 10 queries: ~70-80MB JS heap + 50-60MB WASM
  After 50 queries: ~100MB JS heap + 200MB+ WASM (fragmented)
  Peak possible: 256MB WASM memory (hard limit)

ROOT CAUSES:
  - Serialization creates 3-5x data duplication
  - WASM linear memory never reclaims freed space
  - Pending calls hold references to large data structures
  - TypedArray views can prevent GC of entire WASM memory

================================================================================
MEMORY LEAK SCENARIOS
================================================================================

SCENARIO 1: Repeated API calls
  - Each call creates pending entry
  - Entry deleted on completion (should be)
  - But race condition can prevent deletion
  - Result: ~100 stale entries per incident = 50KB+ wasted

SCENARIO 2: Worker crash
  - Worker dies mid-operation
  - Pending calls never get responses
  - Main thread doesn't detect death
  - New API calls still create pending entries
  - Result: Unbounded accumulation until page reload

SCENARIO 3: Large dataset processing
  - Serialize 500K entries: 50MB string
  - Cache it: 50MB in memory
  - Pass to WASM: 50MB in linear memory
  - Process: 5-10MB temporary allocations
  - Return result: 5MB new TypedArray
  - Total: ~115MB for 50MB input (2.3x amplification)

SCENARIO 4: TypedArray view prevents GC
  - Extend show years: returns Int32Array view
  - If view kept in closure, ENTIRE WASM memory (256MB) stays allocated
  - JS GC can't collect anything
  - Result: 256MB locked until closure released

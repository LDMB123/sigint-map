  CRITICAL ISSUES (Fix Immediately)
================================================================================

ISSUE #2: SERIALIZATION CACHE HASH COLLISION
  File: serialization.ts, line 185
  Severity: CRITICAL (Silent data corruption)
  
  Problem:
    - Arrays with identical first/middle/last elements collide
    - Different data serializes to same cache key
    - WASM receives stale/wrong data without error
    - Example: [song1, DIFFERENT, song3] and [song1, song2, song3] collide
  
  Impact:
    - Wrong statistics calculated
    - Liberation list outdated
    - Data corruption in aggregations
    - Intermittent, hard to debug bugs

  Fix Time: 1-2 hours

ISSUE #5: UNSTABLE TYPEDARRAY VIEWS
  File: serialization.ts, line 480-500
  Severity: CRITICAL (Memory safety)
  
  Problem:
    - viewTypedArrayFromWasm() returns views into WASM memory
    - Views become invalid if WASM memory grows
    - Returning views from async functions is unsafe
    - Can cause crashes or garbage data
  
  Manifestation:
    - extractYearsTyped() returns unstable view
    - extractSongIdsTyped() returns unstable view
    - aggregatePlayCountsTyped() may crash on array access
  
  Fix Time: 1 hour

ISSUE #9: WORKER CODE INCOMPLETE
  File: worker.ts, line 75-110
  Severity: CRITICAL (Performance)
  
  Problem:
    - Worker disabled in config (useWorker: false)
    - Outdated comment claiming wasm-bindgen incompatibility
    - Modern wasm-bindgen DOES support workers
    - Main thread blocks during large computations
  
  Impact:
    - UI freezes (INP violations) during aggregations
    - Cannot parallelize WASM calls
    - Performance regression vs. potential 50-200ms improvement
  
  Fix Time: 2-3 hours (mostly testing)

================================================================================
  HIGH PRIORITY ISSUES
================================================================================

ISSUE #1: WORKER DISABLED
  - Performance regression (50-200ms INP)
  - Related to Issue #9 above

ISSUE #3: DOUBLE JSON.STRINGIFY
  - 100-200ms overhead for large datasets
  - Mitigated by passing strings directly
  - Still affects fallback path

ISSUE #4: MISSING DEALLOC TRACKING
  - No tracking of WASM-allocated resources
  - FinalizationRegistry needed
  - Memory may accumulate over time

ISSUE #7: POLLING CLEANUP INTERVAL
  - 10-second polling creates unnecessary wake-ups
  - Should use per-call timeouts instead
  - Negligible impact on desktop, measurable on mobile

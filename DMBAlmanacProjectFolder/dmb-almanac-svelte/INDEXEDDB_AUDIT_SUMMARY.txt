================================================================================
                        QUERY PERFORMANCE ANALYSIS
================================================================================

Anti-patterns Found:

1. N+1 Pattern: getGuestAppearances (lines 491-506)
   - First query: appearances
   - Second query: bulkGet shows
   - Risk: Multiple async operations, not in transaction
   - Impact: ~5% performance overhead vs. transactional approach

2. Unbounded Queries: topSlotSongsCombined (lines 1673-1680)
   - Query: db.setlistEntries.toArray()
   - Returns: Up to 40,000+ records
   - Risk: 4-8MB in memory at once
   - Impact: GC pauses on devices with <500MB available RAM

3. Filter on Boolean: Songs with isCover filter
   - Pattern: db.songs.filter(s => s.isCover === true)
   - Reason: Boolean index removed in v3 (low selectivity)
   - Impact: Full table scan instead of index lookup
   - Mitigation: Acceptable for boolean with 50% selectivity

4. Missing Transaction Boundaries: Global search
   - Multiple parallel queries in transaction
   - Risk: Inconsistent snapshot across queries
   - Currently: Mitigated by small result set (<20 items)
   - Recommendation: Keep transaction scope

Recommendations:
  1. Add .limit() to all .toArray() queries
  2. Wrap multi-table queries in transactions
  3. Use compound indexes for sequential filtering
  4. Keep read-only for consistent snapshots

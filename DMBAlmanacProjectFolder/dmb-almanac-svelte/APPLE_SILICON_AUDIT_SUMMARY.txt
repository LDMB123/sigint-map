OPTIMIZATION OPPORTUNITIES (Priority Order)
================================================================================

PRIORITY 1: ProMotion 120Hz Optimization (HIGH IMPACT)
──────────────────────────────────────────────────────
Impact: +5-8% animation smoothness on 120Hz displays
Effort: 2-3 hours
Status: Can implement immediately

Current Issue:
  - Animation durations not aligned to 120Hz frame timing
  - Fixed 300ms durations = ~18 frames @ 60Hz but 25 frames @ 120Hz
  - Creates visual unevenness on ProMotion displays

Solution:
  - Detect 120Hz with @media (update: fast)
  - Use 8.33ms frame-aligned durations (120Hz = 8.33ms per frame)
  - Adjust scroll animations to 120Hz timing

Files to Create/Modify:
  CREATE: src/lib/utils/promotionDetection.ts
  CREATE: src/lib/motion/animation-timings.css
  MODIFY: src/routes/+layout.svelte
  MODIFY: src/app.css

Expected Result:
  - 120Hz displays: Animations lock to exact frame timing
  - Smoother perceived animation
  - Better battery life (fewer timing corrections)


PRIORITY 2: Thermal Management (STABILITY)
──────────────────────────────────────────────────────
Impact: +20% thermal stability under load
Effort: 1-2 hours
Status: Can implement immediately

Current Issue:
  - No awareness of CPU thermal state
  - Sustained high load causes thermal throttling
  - App doesn't adapt to throttling conditions

Solution:
  - Detect thermal state via long animation frames (>100ms)
  - Reduce animation intensity when system is hot
  - Disable GPU optimization hints under pressure

Files to Create/Modify:
  CREATE: src/lib/utils/thermalManagement.ts
  MODIFY: src/app.css (add thermal pressure styles)
  MODIFY: src/routes/+layout.svelte

Expected Result:
  - App remains responsive under thermal load
  - Automatic graceful degradation
  - No user-visible stuttering


PRIORITY 3: E-core Aware Scheduling (BATTERY)
──────────────────────────────────────────────────────
Impact: +12-18% battery life, +20% CPU efficiency
Effort: 3-4 hours
Status: Medium complexity

Current Issue:
  - OS scheduler handles P-core vs E-core distribution
  - App can't explicitly hint to use efficiency cores
  - Background tasks sometimes run on expensive P-cores

Solution:
  - Detect M-series chip via navigator.hardwareConcurrency
  - Map task priorities to core types (P=user-blocking, E=background)
  - Break long tasks into E-core-sized chunks

Files to Create/Modify:
  CREATE: src/lib/utils/applesilicon-scheduling.ts
  UPDATE: Search, indexing, data processing components

Expected Result:
  - Background indexing uses E-cores (1/4 power of P-cores)
  - 30-40% battery life improvement over 2-hour session
  - Zero impact on user-facing responsiveness


PRIORITY 4: Metal Texture Caching (VISUALIZATION)
──────────────────────────────────────────────────────
Impact: +30-50% visualization rendering speed
Effort: 3-4 hours
Status: Medium complexity

Current Issue:
  - D3 visualizations render to SVG DOM
  - Metal must rasterize SVG every frame
  - Large visualizations (1000+ nodes) = expensive rasterization

Solution:
  - Cache D3-rendered content as GPU textures
  - Reuse textures when data hasn't changed
  - Only re-render on actual data updates

Files to Create/Modify:
  CREATE: src/lib/gpu/metalTextureCache.ts
  UPDATE: src/lib/components/visualizations/*

Expected Result:
  - Visualization updates 30-50% faster
  - 40% memory reduction for chart data
  - 25% GPU power reduction


PRIORITY 5: WebGPU Rendering (RESEARCH)
──────────────────────────────────────────────────────
Impact: +300-500% for large visualizations
Effort: 5-7 hours (experimental)
Status: Requires Chrome 127+

Current Issue:
  - D3 visualizations limited to SVG rasterization
  - Large graphs (5000+ nodes) = slow rendering
  - GPU not fully utilized

Solution:
  - Implement WebGPU compute shaders for D3
  - Render directly to GPU textures
  - Use Metal backend on Apple Silicon

Files to Create/Modify:
  CREATE: src/lib/gpu/d3WebGPURenderer.ts
  UPDATE: TransitionFlow, GuestNetwork visualizations

Expected Result:
  - Large visualizations render in GPU (Metal)
  - 300-500% faster for 5000+ node graphs
  - Truly interactive graph exploration

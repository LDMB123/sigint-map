ISSUE #1: CACHE COLLISION BUG
├─ File: serialization.ts:190-220
├─ Line(s): 190-220
├─ Type: CORRECTNESS BUG
├─ Pattern: Length-based cache keys
├─ Speedup: 20-50%
├─ Effort: 45 min
├─ Risk: HIGH (potential data corruption)
├─ Problem: getCacheKey uses array.length as key → two different arrays
│           with same length get identical cache key → wrong data returned
├─ Solution: Use WeakMap for object identity instead of length
└─ Status: ⚠️ CRITICAL - Fix immediately

ISSUE #2: JSON STRINGIFY BEFORE CACHE CHECK
├─ Files: queries.ts (15+ functions)
├─ Lines: 63, 103, 151, 190, 239, 316, 374, 420, 473, 527, 608, 659, 718, 775, 832
├─ Type: PERFORMANCE
├─ Pattern: Stringify all args before checking cache
├─ Speedup: 15-40% on cached queries
├─ Effort: 30 min (per function, 10 sec each with macro)
├─ Risk: LOW
├─ Problem: Cache check happens AFTER expensive JSON.stringify
│           If cache hits: 50-200ms wasted per call
│           Affects: wasmGlobalSearch, wasmGetTourStatsByYear, etc.
├─ Solution: Move JSON.stringify inside cache miss branch
└─ Status: 🔴 HIGH PRIORITY - Pattern repeated 15+ times

ISSUE #3: TEXTENCODER/TEXTDECODER NOT CACHED

├─ Speedup: 2-5%

├─ Risk: NONE
├─ Problem: TextEncoder/TextDecoder instances created repeatedly

│           const textDecoder = new TextDecoder();
└─ Status: 🟡 QUICK WIN

ISSUE #5: O(N) LRU CACHE EVICTION
├─ File: serialization.ts:225-250
├─ Lines: 225-250
├─ Type: PERFORMANCE
├─ Pattern: Linear scan for each eviction
├─ Speedup: 5-15%
├─ Effort: 30 min
├─ Risk: LOW
├─ Problem: While loop finds oldest entry by scanning entire cache
│           If cache has 1000 entries and evict 100:
│           100 loops × 1000 scans = 100,000 comparisons
│           O(n*k) where n=cache size, k=evictions needed
├─ Solution: Sort array once, then iterate for eviction
│           O(n log n) sort + O(k) iteration
│           10x faster for typical evictions
└─ Status: 🟡 MEDIUM PRIORITY

ISSUE #6: BINARY ENCODING FOR TYPED ARRAYS
├─ File: bridge.ts:540-570
├─ Lines: 540-570
├─ Type: PERFORMANCE
├─ Pattern: Typed array methods exist but still use JSON
├─ Speedup: 20-40%
├─ Effort: 2 hours
├─ Risk: MEDIUM (requires WASM side coordination)
├─ Problem: getShowIdsForSongTyped() exists but calls JSON.stringify()

└─ Status: 🟡 HIGH VALUE - Medium effort, big payoff

ISSUE #9: .MAP() VS FOR-LOOP OVERHEAD
├─ File: fallback.ts (multiple locations)
├─ Lines: Throughout
├─ Type: PERFORMANCE
├─ Pattern: .map() where value not returned
├─ Speedup: 3-10%
├─ Effort: 1 hour
├─ Risk: NONE
├─ Problem: .map() has ~10% overhead vs for-loops on large arrays
│           1000+ items: noticeable difference
│           Examples: mapping to create side-effect-only transformations
├─ Solution: Use for-of loops where return value not needed
│           Keep .map() where actually creating new array
└─ Status: 🟢 LOW PRIORITY (low ROI)
